# Fineract Client Module Documentation

This document provides a comprehensive overview of the Fineract `portfolio/client` module. This module is central to the Fineract platform, responsible for managing all aspects of a client's (i.e., a customer's) lifecycle and their associated information.

## 1. High-Level Overview

The `org.apache.fineract.portfolio.client` package manages the core client entity and its related components. It is designed following a CQRS-like pattern, where write operations (Commands for create, update, delete) are distinctly separated from read operations (Queries).

The module is organized into several key sub-packages:

-   **`api/`**: Defines the public RESTful API endpoints for interacting with client data. This is the primary interface for external applications.
-   **`data/`**: Contains the Data Transfer Objects (DTOs) used for API requests and responses. These define the JSON structures for communication.
-   **`domain/`**: Includes the core business objects as JPA entities (e.g., `Client`, `ClientCharge`) and their corresponding repositories for database interaction.
-   **`service/`**: Contains the main business logic. `ReadPlatformService` implementations handle data retrieval (often using direct JDBC for performance and flexibility), while `WritePlatformService` implementations handle data modification, business rule validation, and state changes.
-   **`handler/`**: Connects the command processing system to the write services. These handlers are triggered by API calls and invoke the correct business logic based on the requested command.

## 2. Key APIs and Functionality

The module exposes a rich set of REST APIs to manage clients and their related information.

### Core Client Management (`ClientsApiResource.java`)

This is the primary resource for creating, viewing, and managing clients.

-   **`POST /v1/clients`**: Creates a new client. A client can be a person (`legalFormId`: 1) or an entity/business (`legalFormId`: 2). The client can be created in a "Pending" state or directly activated by passing `active: true` and an `activationDate`.
-   **`GET /v1/clients`**: Lists and searches for clients. It supports filtering by office, name, status, etc., as well as pagination and sorting.
-   **`GET /v1/clients/{clientId}`**: Retrieves the complete details of a single client.
-   **`PUT /v1/clients/{clientId}`**: Updates the basic information of an existing client.
-   **`POST /v1/clients/{clientId}?command={action}`**: A versatile endpoint for performing state transitions and other actions on a client:
    -   `command=activate`: Activates a pending client.
    -   `command=close`: Closes a client (requires no active loans or savings accounts).
    -   `command=reject` / `command=withdraw`: Rejects or withdraws a pending client application.
    -   `command=assignStaff`: Assigns a loan officer to the client.
    -   `command=unassignStaff`: Unassigns the current loan officer.
-   **`GET /v1/clients/{clientId}/accounts`**: Provides a consolidated summary of the client's associated loan and savings accounts.

### Client Identifiers (`ClientIdentifiersApiResource.java`)

Manages official identification documents for a client.

-   **`POST /v1/clients/{clientId}/identifiers`**: Adds a new identifier (e.g., Passport, Driver's License) to a client's profile.
-   **`GET /v1/clients/{clientId}/identifiers`**: Lists all identifiers associated with a client.
-   **`PUT /v1/clients/{clientId}/identifiers/{identifierId}`**: Updates an existing identifier.
-   **`DELETE /v1/clients/{clientId}/identifiers/{identifierId}`**: Deletes an identifier.

### Client Charges (`ClientChargesApiResource.java`)

Manages fees and penalties applied directly to a client's account.

-   **`POST /v1/clients/{clientId}/charges`**: Applies a predefined charge to a client.
-   **`GET /v1/clients/{clientId}/charges`**: Lists all charges for a client, with status filtering (all, active, inactive).
-   **`POST /v1/clients/{clientId}/charges/{chargeId}?command={action}`**:
    -   `command=paycharge`: Records a full or partial payment against a specific charge.
    -   `command=waive`: Waives the entire outstanding amount of a charge.

### Client Transactions (`ClientTransactionsApiResource.java`)

Provides a view of financial transactions made against a client's account, which are typically generated by paying or waiving charges.

-   **`GET /v1/clients/{clientId}/transactions`**: Lists all financial transactions for a client.
-   **`POST /v1/clients/{clientId}/transactions/{transactionId}?command=undo`**: Reverses a client transaction, which rolls back the corresponding charge payment or waiver.

### Other Supporting APIs

-   **`ClientAddressApiResource.java`**: Manages one or more physical addresses for a client.
-   **`ClientFamilyMembersApiResource.java`**: Manages information about a client's family members.
-   **`ClientSearchV2ApiResource.java`**: A dedicated endpoint for performing full-text searches on clients.

## 3. Core Data Structures

These are the primary data objects used in API requests and responses.

### `ClientData`

The main DTO representing a client. It is a comprehensive object containing:
-   **Basic Info**: `id`, `accountNo`, `externalId`, `status`, `firstname`, `lastname`, `displayName`, `dateOfBirth`, `gender`.
-   **Organizational Info**: `officeId`, `officeName`, `staffId`, `staffName`.
-   **Status & Timeline**: `activationDate`, and a `timeline` object that tracks submission, activation, and closing events (dates and users).
-   **Associations**: `groups` (if the client is part of a group), `savingsAccountId` (for the default savings account).
-   **Non-Person Details**: If the client is an entity, the `clientNonPersonDetails` object holds information like incorporation number and constitution.
-   **UI Template Data**: When requested with `template=true`, the object is populated with `officeOptions`, `staffOptions`, `genderOptions`, etc., which are useful for building user interface dropdowns.

### `ClientIdentifierData` & `ClientIdentifierRequest`

-   **Structure**: Contains `documentTypeId` (a system-defined code value for the type of ID), `documentKey` (the unique ID number, e.g., passport number), and a `description`.
-   **Usage**: Used to represent, create, and update client identification records.

### `ClientChargeData`

-   **Structure**: Represents an instance of a charge applied to a client. Includes `chargeId`, `name`, `dueDate`, `amount`, `amountPaid`, `amountWaived`, and `amountOutstanding`.
-   **Usage**: Returned when listing client charges and is the basis for making payments or waivers.

### `ClientTransactionData`

-   **Structure**: Represents a financial event. Includes `id`, `type` (e.g., "Pay Charge"), `date`, `amount`, and `reversed` status.
-   **Usage**: Provides a read-only history of financial activities related to the client's account.
