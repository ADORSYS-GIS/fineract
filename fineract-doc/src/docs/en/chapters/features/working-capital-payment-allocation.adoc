= Working Capital Payment Allocation

== Concept
On legacy and merchant term loans (progressive loans), 3 timely modifier were defined:

* PAST DUE
* DUE
* IN ADVANCE

For sales-based Working Capital loan products, PAST DUE is not applicable, but DUE and IN ADVANCE balances provide enough flexibility and fine-grained configuration options for business needs.

* *DUE*:
** *Principal*: Projected principal payment
** *Fee* and *Penalty*: current or older charges (based on specific due date)
* *IN ADVANCE*:
** *Principal*: over projected principal payment
** *Fee* and *Penalty*: Future charges (based on specific due date)

== Validation

Working Capital Loans accept the following `paymentAllocationRule` enum values. Each payment allocation order must include all six values exactly once.

* `DUE_PRINCIPAL`
* `DUE_FEE`
* `DUE_PENALTY`
* `IN_ADVANCE_PRINCIPAL`
* `IN_ADVANCE_FEE`
* `IN_ADVANCE_PENALTY`

During loan product creation, at least one `DEFAULT` transaction type allocation must be provided. Additional transaction-type-specific allocations can be provided, but each transaction type can appear only once.

== Loan Product Creation & Update

During loan product creation, `paymentAllocation` is mandatory. At minimum, provide one allocation with `transactionType = "DEFAULT"`.

During loan product update, payment allocation rules are maintained by `transactionType`: existing transaction types are updated, new transaction types are created, and transaction types omitted from the incoming allocation list are removed.

=== Template

Template API for Loan Product related operations are providing the available payment allocation settings.

== Loan Account Creation & Update

During loan account creation, `paymentAllocation` is optional. When provided, it is used for the account. If not provided, the loan product's settings are copied to the new loan account. This keeps account-level payment allocation independent from later product-level changes.

During loan account update, payment allocation rules are also maintained by `transactionType`: matching existing transaction types are updated, new ones are created, and omitted existing transaction types are deleted.
