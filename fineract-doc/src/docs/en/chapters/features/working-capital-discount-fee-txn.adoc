== Working Capital Loan — Discount Fee

=== 1. Overview

The **Discount Fee** is applicable for **Working Capital Loan** products and represents an upfront charge applied on the disbursed amount.

Instead of collecting interest over time, the loan applies a **discounting mechanism**, where:
* The borrower receives a **net disbursed amount** (principal minus discount fee)
* The **full principal amount** remains repayable

=== 2. Applicability

| Condition | Status |
|----------|--------|
| Loan Type: Working Capital Loan | ✔ Supported |
| Loan Type: Progressive Loan | ✘ Not supported |
| Loan Type: Cumulative Loan | ✘ Not supported |

* Discount Fee functionality is **exclusively available** for Working Capital Loan products.
* It is not applicable for Progressive or Cumulative loan types.


=== 3. Core Concept

The Discount Fee represents an upfront charge applied on the loan and affects the **net disbursed amount**, while the **full principal remains repayable**.

The Discount Fee can be configured and applied at multiple stages of the loan lifecycle:

* At **Loan Application**
* At **Loan Approval**
* At **Disbursement**
* **After Disbursement**, only if it is the **first monetary transaction following disbursement**

This flexibility allows financial institutions to align discounting behavior with their operational and approval workflows.

=== 4. Accounting & GL Handling

⚠ Accounting integration for Discount Fee is **not implemented yet**.

=== 5. Transaction Handling

* A **Discount Fee Transaction** is created:
- During **disbursement**, if discount fee is already defined OR provided in disbursement request
* The transaction is created as a **separate transaction**
* Each **disbursement** can have **only one associated Discount Fee transaction**

Relationship rules:
* Discount Fee transaction is always **linked to a disbursement transaction**
* Link is maintained via `relatedResourceId`

Schedule & balance updates:
* Loan balances are updated **immediately** after transaction creation
* Amortization Schedule is updated **immediately**
* Allocation for Discount Fee transaction is applied to **principal**

=== 6. Reversal Behavior

* Disbursement reversal is allowed only if:
- No other monetary transactions exist on the loan
- No additional disbursements exist

* When a disbursement is reversed:
- Its associated Discount Fee transaction must also be **reversed**
- Reversal maintains consistency between:
- Loan balance
- Amortization Schedule

* Discount Fee transaction cannot exist without its related disbursement

=== 7. Discount Fee Adjustment

A **Discount Fee Adjustment** reduces the loan-level discount that was established by an existing, non-reversed **Discount Fee** transaction. Behaviour is aligned with other *adjustment* transaction types: linked parent transaction, allocation row, schedule restate, and business event.

==== Purpose

* Correct a discount fee amount
* Recalculate EIR and amortisation schedule

==== Applicability

* **Working Capital Loan** only (same as Discount Fee)
* Loan must be **Active**
* Parent transaction must be an active (non-reversed) **Discount Fee** transaction

==== Transaction behaviour

* Creates a separate transaction of type `DISCOUNT_FEE_ADJUSTMENT`
* Linked to the original Discount Fee transaction via `relatedResourceId` (in the request body) and a `RELATED` loan transaction relation
* **Multiple adjustments** are allowed against the same Discount Fee transaction until the remaining adjustable amount is zero
* Reduces `loanProductRelatedDetails.discount` by the adjustment amount

==== Validation rules

[cols="2,3"]
|===
| Rule | Detail

| Amount
| Mandatory; must be **> 0**

| Maximum amount
| Must not exceed *remaining discount* = original discount fee amount minus sum of prior non-reversed adjustments

| Transaction date
| Optional in request; defaults to the parent Discount Fee date if omitted

| Date vs discount fee
| Must **not** be before the parent Discount Fee transaction date

| Backdating
| **Not allowed** — transaction date must be on or after the current business date

| Loan status
| Adjustment allowed only for **Active** loans

| Parent transaction
| `relatedResourceId` must reference a non-reversed **Discount Fee** transaction
|===

==== Allocation

* An allocation row is stored in `m_wc_loan_transaction_allocation`
* Adjustment amount is allocated entirely to **principal portion** (same pattern as Discount Fee)

==== Schedule impact

On adjustment the system:

. Regenerates the projected amortization schedule using the updated loan discount and disbursement data
. Re-applies all recorded **actual repayments** (preserved from the previous model)
. Updates loan balances

==== Business event

* `WorkingCapitalLoanDiscountFeeAdjustmentTransactionBusinessEvent` — posted after a successful adjustment

== API Endpoints

* *Endpoint*: `POST /v1/working-capital-loans/{loanId}/transactions?command=discountFeeAdjustment`
* *Alternative*: `POST /v1/working-capital-loans/external-id/{loanExternalId}/transactions?command=discountFeeAdjustment`

[source,json]
----
{
  "transactionAmount": 50.0,
  "relatedResourceId": 99,
  "transactionDate": "10 January 2026",
  "dateFormat": "dd MMMM yyyy",
  "locale": "en",
  "note": "Reduce discount fee",
  "externalId": "WC-DISC-ADJ-001"
}
----

* `relatedResourceId` — **Discount Fee** transaction id (not disbursement id)
* `transactionAmount` — amount to reduce from the current loan discount (capped by remaining adjustable discount)
